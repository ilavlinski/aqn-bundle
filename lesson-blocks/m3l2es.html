Semana 3: Programación Orientada a Objetos para Aeroespacial

POO significa organizar el código en objetos reutilizables. En lugar de escribir funciones dispersas, defines clases que agrupan datos y comportamiento juntos. En la aeroespacial, así es como pensamos sobre los sistemas: una nave espacial es un objeto con propiedades (masa, altitud, velocidad, combustible) y comportamientos (quemar motores, comunicarse a casa, ajustar actitud). Un cohete es un objeto con empuje, masa, combustible, y la capacidad de calcular aceleración y delta-v.

Semana 4: NumPy y Pandas para Datos Aeroespaciales

NumPy proporciona matrices rápidas y operaciones matemáticas. Pandas proporciona datos tabulares (DataFrames) que es cómo se organiza la telemetría real. Cada ingeniero aeroespacial usa estas bibliotecas para analizar datos de misiones.

Semana 5: APIs REST y Datos en Vivo

Las APIs (Interfaces de Programación de Aplicaciones) permiten que tu código solicite datos a servidores. La NASA proporciona APIs gratuitas. SpaceX proporciona una API. N2YO proporciona seguimiento de satélites en tiempo real. Usa la biblioteca requests para obtener datos en formato JSON, luego analízalos con Python.

Semana 6: Visualización de Datos con Matplotlib

Los datos sin procesar son difíciles de entender. Los gráficos cuentan la historia. Traza trayectorias orbitales, pases de satélites, ventanas de lanzamiento, tendencias de telemetría.

Programación Orientada a Objetos: Un Ejemplo Real

Antes de que alguien te pida escribir POO en una entrevista técnica, practícalo en algo que te importe. Aquí está una clase Rocket simplificada. Llévala al notebook y ejecútala. La versión completa con múltiples etapas, arrastre y gravedad inversa-cuadrada vive en aqn_oop_rocket_class.py en los descargables del módulo — 150 líneas, cada una comentada.

class Cohete:
    """Un cohete de una sola etapa con combustible y empuje constantes."""

    G0 = 9.80665  # gravedad estándar, m/s^2

    def __init__(self, masa_seca, masa_propelente, empuje, isp):
        self.masa_seca = masa_seca
        self.masa_propelente = masa_propelente
        self.empuje = empuje
        self.isp = isp
        self.t = 0.0
        self.v = 0.0
        self.h = 0.0

    @property
    def flujo_masico(self):
        return self.empuje / (self.isp * self.G0)

    @property
    def masa_actual(self):
        quemado = self.flujo_masico * self.t
        return self.masa_seca + max(0.0, self.masa_propelente - quemado)

    def paso(self, dt):
        t_efectivo = self.empuje if self.flujo_masico * self.t < self.masa_propelente else 0.0
        a = t_efectivo / self.masa_actual - self.G0
        self.v += a * dt
        self.h += self.v * dt
        self.t += dt


if __name__ == "__main__":
    cohete = Cohete(masa_seca=500, masa_propelente=1500, empuje=40000, isp=260)
    for _ in range(200):
        cohete.paso(0.5)
    print(f"t={cohete.t:.0f}s  altura={cohete.h/1000:.1f}km  v={cohete.v:.0f}m/s")

Lo que enseña este ejemplo: las clases encapsulan estado (masa, velocidad, altura) y comportamiento (paso). Las propiedades (@property) calculan valores derivados bajo demanda. El método paso es un integrador Euler simple. Cuando lees código Python real en un trabajo, verás este patrón en todas partes.

Consumo de APIs REST: El Patrón que Necesitas Dominar

En tu primer trabajo como ingeniero de datos o de software en aeroespacial, harás esto cientos de veces: llamar a un API, parsear JSON, manejar errores, respetar límites de tasa. Aquí está el patrón completo en ~40 líneas.

import time
import requests
from typing import Any

class ApiCliente:
    def __init__(self, base_url: str, api_key: str, max_reintentos: int = 3):
        self.base_url = base_url.rstrip("/")
        self.session = requests.Session()
        self.session.headers.update({"Authorization": f"Bearer {api_key}"})
        self.max_reintentos = max_reintentos

    def obtener(self, ruta: str, params: dict = None) -> dict:
        url = f"{self.base_url}/{ruta.lstrip('/')}"
        for intento in range(self.max_reintentos):
            r = self.session.get(url, params=params, timeout=30)
            if r.status_code == 429:  # límite de tasa
                espera = int(r.headers.get("Retry-After", 2 ** intento))
                time.sleep(espera)
                continue
            if r.status_code >= 500:
                time.sleep(2 ** intento)  # backoff exponencial
                continue
            r.raise_for_status()
            return r.json()
        raise RuntimeError(f"Falló después de {self.max_reintentos} intentos: {url}")


# Ejemplo: Celestrak API para TLEs de satélites
cliente = ApiCliente("https://celestrak.org", api_key="")
tle_iss = cliente.obtener("NORAD/elements/gp.php", {"CATNR": "25544", "FORMAT": "TLE"})

Cinco detalles importantes aquí: sesión persistente (reusa conexiones TCP), autenticación en headers (nunca en URL), reintentos con backoff exponencial, manejo explícito de 429, y timeouts. Cuando un reclutador te pregunta "¿cómo consumes un API?", muestras este patrón y pasas la primera ronda técnica.

Git Workflow Profesional: La Secuencia de 7 Pasos

En la universidad te enseñaron git commit -am "arreglo". Ese no es el Git del trabajo. Aquí está el workflow que se espera en cualquier equipo senior de aeroespacial o NewSpace.

  1. git checkout main && git pull — empieza limpio desde main actualizado
  2. git checkout -b feature/AQN-142-satellite-parser — rama con ID de ticket
  3. Haz cambios pequeños y atómicos. Un commit, un concepto.
  4. git add -p — revisa cada cambio interactivamente antes de commit
  5. git commit -m "Parse Celestrak TLE in streaming mode (AQN-142)" — mensaje en imperativo, con ID
  6. git push origin feature/AQN-142-satellite-parser — empuja la rama
  7. Abre un Pull Request con descripción, capturas, y checklist. Espera revisión de al menos un compañero antes de mergear.

Este patrón resuelve el 90% de los problemas de colaboración: quién cambió qué, por qué, cuándo, y quién lo revisó. Los equipos que no hacen esto son los que a las 2 AM del jueves tienen un incidente en producción porque nadie sabe qué cambio lo causó.
