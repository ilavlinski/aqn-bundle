Why GitHub Is the Hiring Signal

At NASA Goddard, JPL, SpaceX, Blue Origin, Rocket Lab, and Satellogic, recruiters do the same thing: They search GitHub for engineers who can code. They look at commit history. They check if a developer writes clean code. They see if projects are documented. They want to know if you collaborate, if you handle feedback, if you iterate. A GitHub profile is a resume written in code. It is the most honest thing you can show an employer.

Katya Echazarreta, who worked at NASA JPL and flew on Blue Origin New Shepard, has a public GitHub. Emiliano Kargieman, who founded Satellogic and took it public, has public repositories showing his path as a founder-engineer. The Tobacia brothers at Blue Origin contribute to open-source projects. They understand that your GitHub is your professional brand.

Five Portfolio Projects with Complete Specifications

Project 1: Satellite Tracker Web App (Real-Time TLE Data)

  • Tech stack: Python backend (Flask or FastAPI), HTML/CSS/JavaScript frontend, real-time TLE data from N2YO or NASA
  • Features: Search satellite by name, display current position on interactive map (Leaflet.js), show orbit prediction for next 24 hours, alert when satellite passes overhead
  • Data source: N2YO API or Space-Track.org TLE data, Open-Meteo for weather integration
  • Deliverables: GitHub repo, deployed web app (free tier on Heroku or Vercel), README with setup instructions and architecture diagram

Project 2: Rocket Equation Simulator with GUI (Real Physics)

  • Tech stack: Python with Tkinter GUI or PyQt, NumPy for calculations, Matplotlib for plots
  • Features: Input rocket parameters (mass, thrust, ISP, fuel), calculate delta-v, plot acceleration/altitude/velocity profiles over time, compare different rocket designs side-by-side
  • Real physics: Gravity losses, drag, actual rocket equation (not simplified)
  • Deliverables: GitHub repo, executable app, README with physics documentation and screenshots

Project 3: NASA Image of the Day CLI Tool (Learning APIs)

  • Tech stack: Python, requests library, argparse for command-line interface
  • Features: Fetch NASA Astronomy Photo of the Day, download image, display metadata (date, title, description), search by date range, save image locally or upload to cloud
  • Stretch goal: Create a web gallery of saved images with filtering and search
  • Deliverables: GitHub repo with detailed README, pip-installable package, documentation

Project 4: ISS Pass Predictor for Your City (End-to-End)

  • Tech stack: Python backend, web or CLI interface
  • Features: Enter your city name or coordinates, get ISS pass times for next 7 days, show rise time, max altitude, set time, brightness, direction of approach
  • Data: Real TLE data from NASA or N2YO, real ISS position calculations
  • Advanced features: Notifications (email/SMS when ISS is 2 hours away), historical database of past passes
  • Deliverables: GitHub repo, live web version, README, video demo

Project 5: Orbital Mechanics Visualizer (3D, Interactive)

  • Tech stack: Python with Matplotlib 3D or JavaScript with Three.js, NumPy for orbit propagation
  • Features: Visualize Earth, Moon, ISS, geostationary satellites, user-defined orbits in 3D. Animate orbital motion. Show gravity vectors. Allow interactive camera movement.
  • Real physics: Use SGP4 propagation model (pyorbital or skyfield library)
  • Deliverables: Interactive visualization (web version preferred), GitHub repo with documentation, video walkthrough

README Best Practices: What Gets You Hired

Essential Sections (In This Order)

  1. Project Title and One-Liner - What does it do in one sentence. Example: "Real-time ISS position tracker with orbital prediction and web interface."
  2. Badges - Status, license, build status, language. Example: ![Python](https://img.shields.io/badge/Python-3.9+-blue) ![License](https://img.shields.io/badge/License-MIT-green)
  3. Demo or Screenshot - Show, do not tell. Include a GIF or screenshot. If web app, include live link.
  4. Table of Contents - For longer READMEs, helps readers navigate.
  5. Features - Bullet list of what the project does. Focus on user value, not implementation.
  6. Getting Started (Install + Run) - Step-by-step: clone repo, create virtual environment, pip install requirements, run the app. Must work exactly as written.
  7. Usage Examples - Code snippets or command-line examples showing how to use the tool.
  8. Architecture / How It Works - Diagram or text explanation of the system design. Aerospace projects especially need this: what data flows where, what are the key algorithms.
  9. API Documentation - If you exposed functions or endpoints, document them (parameters, return values, errors).
  10. Testing - How to run tests. What tests exist. Coverage percentage if available.
  11. Contributing - How others can report bugs or submit code changes.
  12. License - MIT, Apache 2.0, or GPL. Aerospace companies prefer permissive licenses (MIT, Apache).
  13. Acknowledgments and References - Credits to APIs, libraries, papers, people who helped. Shows you collaborate.

Version Control Workflow: How Professional Teams Collaborate

You have written code. Now learn how teams work together. Every aerospace company uses this workflow:

  1. Create a branch for each feature: git checkout -b feature/iss-predictor-alerts
  2. Make commits with clear messages: "Add email notification for ISS passes" (not "fix stuff")
  3. Push to GitHub: git push origin feature/iss-predictor-alerts
  4. Open a Pull Request (PR): Write a description of what you changed and why. Link any related issues.
  5. Address feedback: Respond to code review comments, make improvements, commit again.
  6. Merge when approved: Branch gets merged into main. Feature is live.

This process ensures code quality, team communication, and a reviewable history. Every line of flight-critical software has a PR behind it.

Your GitHub is your reputation. Build it like it matters. Because it does.

Answer Key and Rubric

Question 1 (Complete README): Must include all sections. Title and one-liner clear and concise. Badges present (language, license, etc.). Features bullet list covers core functionality. Installation instructions are step-by-step and tested. Usage examples include code snippets or commands. Architecture explanation is clear to someone unfamiliar with the project. License is specified (MIT or Apache recommended for aerospace). Minimum 400 words. Professional tone throughout.

Question 2 (Commit log): Each commit message starts with an action verb ("Add", "Fix", "Improve", etc.). Messages are specific (not "update code"). Log tells a coherent development story. Commits seem to represent logical, reviewable units of work. Shows understanding of professional development workflow.

Question 3 (Profile optimization): Bio is specific to aerospace/coding, not generic. Shows understanding of what recruiters search for. Six pinned projects are your best work with strong READMEs. Commit consistency strategy is realistic and measurable (e.g., "Code 4 days/week, 10+ commits/month"). Shows awareness of GitHub as a professional tool.

Question 4 (Architecture explanation): Identifies main components (backend, frontend, APIs, databases, etc.). Shows data flow clearly (input sources, processing, outputs). Lists dependencies explicitly (libraries, APIs, datasets). Explains why components are organized this way, not just what they are. Diagram (if provided) is clear and labeled. Minimum 300 words or clear, professional diagram.

Question 5 (Contribution guide): Development setup instructions are complete (Python version, virtual env, pip install, required API keys). Pull request template is defined (title format, description requirements, testing checklist). Code standards are explicit (style guide reference like PEP 8, commenting requirements, test coverage). Code review process is defined (who reviews, timeline, feedback expectations). Shows maturity and collaboration mindset.

Question 6 (Company/requirement mapping): Student correctly identifies a real aerospace company and a real job requirement. Describes specifically which project(s) demonstrate that skill. Examples are concrete, not vague. Shows understanding of how a portfolio builds credibility with real employers.

Question 7 (Reflection): Genuine reflection on learning. Mentions specific skills or mindsets developed (collaboration, documentation, iterating on feedback, shipping code). Shows growth from "I can code" to "I can engineer". Connects portfolio work to real aerospace engineering practices.
